# Ship Aether as a self-contained, instantly-iterable container

<RichText id="overview">

### Objective

Package the Aether world so it boots into a finished experience — autologin straight into the familiar, an RPG-flavoured terminal, and a full-screen character sheet with an ASCII portrait — while keeping the day-to-day edit/test loop **instant**: change a Python file on the host and re-run `./play` in the running container, with **no image rebuild, ever**.

"Done" means three things hold:

1. `make dev` boots a `systemd-nspawn` container that autologins as `player` and drops straight into the Glade; the host's `~/aether` is bind-mounted live, so editing `engine/familiar.py` on the host and re-running `./play` inside shows the change immediately.
2. Typing `character` opens a full-screen `curses` sheet — ASCII portrait on the left, live stats/spellbook/deeds on the right, the wraith's state and XP bar updating in real time — and `q` returns to the world.
3. `make image` bakes the current game into a copy of the golden rootfs and emits a single `dist/aether.tar.zst` — the shippable "one unit" — which boots with a writable overlay that doubles as the save/permadeath system.

</RichText>

<RichText id="key-insight-heading">

### The architecture that removes the rebuild

</RichText>

<RichText id="key-insight">

The pain you're avoiding — "recompress an ISO to test a change" — disappears once two artifacts that change at completely different rates are decoupled. The **golden base rootfs** (OS + terminal packages + autologin) changes rarely. The **game code** (`~/aether`) changes constantly. During development the game never lives *inside* the image: it's bind-mounted from the host, so the image is static and the code is live. The image is rebuilt only when a platform-level thing changes (a package, a default theme), and even then it's one `make` target, not a burn.

</RichText>

<Callout id="decision-runtime" tone="decision">

**Runtime: `systemd-nspawn` container, not a VM.** It gives the full "OS-as-engine" model the game depends on — real PID 1, real processes (`river.py`/`wraith.py`), real permissions and signals — but boots in well under a second and rebuilds in one. Snapshots come from overlay/btrfs instead of qcow2. A full QEMU/KVM VM would add minutes of boot+image overhead per iteration for isolation this single-player game doesn't need.

</Callout>

<Callout id="decision-tui" tone="decision">

**Character sheet: stdlib `curses`, not Textual.** Zero dependencies — it ships clean in the rootfs with nothing to `pip install`, and it renders perfectly on the bare container console. A full-screen ASCII-portrait + live-stats sheet is well within curses' reach; the only cost is slightly more manual layout code, which is a one-time write.

</Callout>

<RichText id="decision-distro">

**Base distro: Arch (`pacstrap`).** It matches your host, you already run it, and — load-bearing for this game — its GNU coreutils are exactly the verbs the world assumes (`chmod`, `tail -f`, `ln -s`, `ps`, `kill`). Alpine's BusyBox variants differ in flags and would quietly change puzzle behaviour. The base meta-package pulls no kernel, which is correct for a container.

</RichText>

<RichText id="scope-heading">

### Scope and non-goals

</RichText>

<RichText id="scope">

**In scope:** the container build/run tooling (`container/` + a top-level `Makefile`), autologin + banner + theme config, the new `engine/sheet_tui.py` and `engine/art/`, a one-command release tarball, and the overlay-based save/permadeath model.

**Non-goals (deferred):** a graphical desktop or windowed terminal emulator (the bare console is the surface); multi-player / networked realms (the README's `nc` remote realm stays future work); user-namespaced UID isolation during dev (off by default — see Risks); publishing the tarball anywhere. The game's own logic (`progress.py`, `init_world.py`, the daemons) is **not** rewritten — only `familiar.py` gains one branch to launch the TUI.

</RichText>

<RichText id="phase1-heading">

### Phase 1 — Host prerequisites and the golden base rootfs

</RichText>

<RichText id="phase1">

`systemd-nspawn` and `machinectl` already ship with `systemd` on Arch; the one extra host package is `arch-install-scripts` for `pacstrap`. The base is built once into a project-local directory (`base/`, git-ignored) so nothing pollutes system paths until you choose to register it.

The package list stays deliberately small — the game's point is that real coreutils *are* the spells, so terminal garnish is a thin layer: `figlet` for the title card and a rainbow pipe for the banner. App-level ANSI theming (your existing `embertheme`) does the heavy lifting so the look survives on any console.

</RichText>

<RichText id="phase2-heading">

### Phase 2 — RPG terminal: autologin → straight into the world

</RichText>

<RichText id="phase2">

An nspawn `--boot` container runs systemd as PID 1 and presents a login on `/dev/console`, served by `console-getty.service`. A drop-in makes that console autologin as `player`; `player`'s `.bash_profile` prints the banner once and `exec`s `./play`, guarding against re-entry so a nested shell doesn't loop. Boot the container and you are standing in the Glade with no prompt.

</RichText>

<RichText id="phase3-heading">

### Phase 3 — The full-screen curses character sheet

</RichText>

<RichText id="phase3">

`engine/sheet_tui.py` reuses everything `progress.py` already exposes — `current_deeds()`, `xp_from()`, `bar()`, `owned_spells()`, `spell_desc()`, `owned_regalia()`, `level_for()` — so the TUI is pure presentation with no new game logic. It draws a two-pane full-screen layout: an ASCII portrait (loaded from `engine/art/`, chosen by current level/class) on the left, and live stats — XP bar, attributes, spellbook with self-documented descriptions, regalia, deeds, and current wraith mood — on the right. A short input timeout makes it re-read state every second, so calming the wraith or routing the river updates the sheet in real time. `q` exits back to the familiar.

In `familiar.py`, the existing `character` command branches: on a real tty it calls `curses.wrapper(sheet_tui.run)`; otherwise (piped/non-interactive, as in the smoke tests) it falls back to the current inline `character()` text sheet. `char` and `me` stay mapped to the quick inline sheet for a no-clear glance.

</RichText>

<RichText id="phase4-heading">

### Phase 4 — The dev loop (the part that never rebuilds)

</RichText>

<RichText id="phase4">

`make dev` boots the rootfs with the host repo bind-mounted at `/opt/aether`. Because the world is plain Python read fresh on each `./play`, "test a change" is just: edit on the host, then re-run `./play` inside — or wrap it with `watchexec -r ./play` to auto-relaunch on save. Nothing is compiled, packed, or rebuilt.

One consequence to know: during dev, the game's own writes (`world/`, `state/`) land on the **host** repo through the bind mount — which is exactly what you want while building (persistent, inspectable, git-ignored already). The throwaway/permadeath behaviour belongs to the *release* run in Phase 5, not the dev loop.

</RichText>

<Code
  id="phase4-cmd"
  language="bash"
  filename="Makefile (dev target)"
  caption="Boot the container with the live game bind-mounted. uid parity keeps chmod gameplay honest."
  code={"dev:\n\tsudo systemd-nspawn -b -D base/ \\\n\t  --bind=$(HOME)/aether:/opt/aether \\\n\t  --hostname=aether\n# Edit ~/aether on the host -> inside the container: ./play  (or: watchexec -r ./play)\n# No --private-users: container uid 1000 == host uid 1000, so file ownership\n# and permission puzzles behave identically on both sides of the mount.\n"}

<RichText id="phase5-heading">

### Phase 5 — Release as one unit, and the permadeath overlay

</RichText>

<RichText id="phase5">

`make image` copies the current `~/aether` into a build copy of the rootfs at `/opt/aether` and tars it to `dist/aether.tar.zst` — a single file someone can `machinectl import-tar` and boot. That's the shippable "one unit."

The save/permadeath model the README sketches falls out of nspawn's overlay options for free. The shipped image **defaults to a persistent overlay**, so an ordinary boot keeps your progress across reboots — the gentler default — and permadeath is a deliberate opt-in:

- **Default (persistent) run** — `make play-image` boots the immutable base under a persistent overlay directory; looted chests, learned spells, and a calmed wraith survive shutdown. Copy the overlay (or `machinectl clone`) for a save slot.
- **Permadeath run** — pass `--volatile=overlay` to boot under a tmpfs overlay instead; every write is discarded on shutdown, so death = reboot and the world resets pristine. `rm -rf` the persistent overlay to die for real.

This is why the artifact is a `tar.zst`/rootfs and not an ISO: the writable overlay *is* the game state, and it's the same mechanism that makes dev never need a rebuild.

</RichText>

<RichText id="risks-heading">

### Risks and how they're handled

</RichText>

<Callout id="risk-uid" tone="warning">

**Permission parity across the bind mount.** The whole game is built on real `chmod` semantics, so the container's `player` must be **uid 1000** to match the host owner of `~/aether`, and dev must run **without** `--private-users` (which would remap uids and break ownership on the mount). Release images can opt into user-namespacing for isolation since they bake the code in. This is called out in `build.sh` and the `dev` target.

</Callout>

<Callout id="risk-root" tone="risk">

**`pacstrap` and `systemd-nspawn -b` need root.** `make base`, `make dev`, and `make image` use `sudo`. This is expected for container tooling; the targets are explicit about it. No rootless path is attempted in this cut.

</Callout>

<Callout id="risk-lolcat" tone="info">

**The rainbow pipe is optional garnish.** `lolcat` lives in the AUR; the banner script degrades gracefully to `toilet --gay` and then to plain `figlet` if neither is installed, so a fresh `pacstrap` never produces a broken login. The RPG *feel* comes mostly from app-level ANSI theming, which has no package dependency.

</Callout>

<RichText id="verify-heading">

### Verification — an end-to-end smoke

</RichText>

<RichText id="verify">

Beyond byte-compiling the new module, the acceptance run exercises the real journey:

1. `make base` then `make dev` — the container boots and **autologins straight into the Glade** (no login prompt), confirming Phases 1–2.
2. Inside: `character` opens the **full-screen curses sheet** with the ASCII portrait; route the river (`ln -sf ../river world/glade/wraith/feed`), watch the wraith mood and XP bar **update live** on the sheet, then `q` back — confirming Phase 3 and that the live-state deed model drives the TUI.
3. On the host, edit a string in `engine/familiar.py`, then re-run `./play` inside the **same** running container — the change shows **without any rebuild**, confirming Phase 4.
4. `make image` then `make play-image` (persistent default); loot the chest, reboot, and assert it is **still looted** — progress survives. Then reboot once with `--volatile=overlay` and assert the chest is **barred again** — confirming both the persistent default and the permadeath opt-in, and that the tarball is a self-contained unit.

</RichText>

<RichText id="decisions-settled-heading">

### Decisions settled

</RichText>

<RichText id="decisions-settled">

The three direction calls are made and baked into the plan above: the golden base rootfs lives in a **project-local, git-ignored `base/`** (self-contained in `~/aether`, nothing in system paths, trivial to nuke and rebuild); the shipped image **defaults to a persistent overlay** so progress survives reboots, with permadeath as an opt-in `--volatile=overlay` flag; and the first cut ships **one ASCII portrait** (`apprentice.txt`) with the per-level lookup wired but the remaining portraits deferred. No open questions remain — the plan is ready to build on approval.
